Designed to scale to 1000 customers without rework. Different audience than AgentVolt: developers who want their omp-deck always-on, on a tier of compute they don't have to maintain.

1. Summary

omp-deck Managed Hosting sells a turnkey, always-on omp-deck to developers who would rather pay $12–49/mo than self-host the same MIT-licensed stack. Each customer gets a dedicated VM running the deck (Bun + Hono server, React + Vite web, Telegram bridge supervised in-tenant), plus the AgentVolt V0 chassis (control plane, proxy farm, snapshot pipeline, networking posture) underneath.

V0 deliberately ships without an inference gateway. Customers bring their own Anthropic / OpenRouter / OpenAI key; the tenant VM calls vendors directly. This removes the largest single subsystem from AgentVolt's V0, removes all metered-billing complexity, and aligns the product with what developers actually want (their own vendor account, predictable subscription, no markup ambiguity).

The exit path is part of the product, not a Phase-1 backlog item. A "Download Volume Snapshot" button in the dashboard, paired with a documented self-host runbook, makes the switching cost from hosted-to-self-host genuinely low. That is the convenience-pricing thesis: we earn the subscription every month or the customer leaves with their data intact.

Key decisions: No inference gateway in V0; BYOK is default-not-opt-in. Per-tenant Hetzner VMs with the AgentVolt V0 isolation posture (Cloud Firewall + Tailscale ACLs + cgroups). Flat subscription tiers LITE $12 / PRO $25 / POWER $49. MIT-licensed source + dashboard snapshot export + documented self-host runbook constitute a first-class exit path. Auth-bridge sidecar pattern reused verbatim from AgentVolt §5.2.

2. Goals & constraints

Functional

  • Customer signs up via dashboard, pays via Stripe, lands in a live omp-deck in <60s
  • Customer brings their own Anthropic / OR / OpenAI key during onboarding; dashboard validates with a test call
  • Customer opens deck via single dashboard click; never types a deck password
  • Tasks DB, routines, inbox, sessions, skills, knowledge base, marketplace state, Telegram bridge state survive any infrastructure event (restart, upgrade, repair, force-push)
  • Telegram (and future Discord/Slack) bridges configurable from dashboard; supervised in-tenant by the deck's BridgeSupervisor
  • "Download Volume Snapshot" works end-to-end: encrypted tar + customer-password-wrapped DEK + runbook for docker compose up

Operational

  • Once provisioned, ops does not touch a customer's VM. Auto-repair handles failures unattended.
  • Customer-pinned versions respected indefinitely. Force-push reserved for CVE-class events, audit-logged.
  • Repair flow: <90s recovery (realistic 60–90s for VM recreate; matches AgentVolt §5.5)
  • Daily R2 snapshot per tenant; pre-upgrade snapshot before any version change.

Economic

  • Three tiers: LITE $12 / PRO $25 / POWER $49, all net-profitable from customer #1
  • Per-tenant infra cost: ~$5.62 LITE / ~$6.55 PRO / ~$12.10 POWER (VM + sized volume, no IPv4) — identical to AgentVolt
  • Fixed ops cost target: ~$110–130/mo at 1000 customers (excluding R2 storage), down from AgentVolt's $160 because no gateway pair, no reconciler
  • Floor case (1000 customers, 50/40/10 LITE/PRO/POWER split): ARR ~$251k, gross ~$171k, net ~$170k/yr
  • Realistic case (30/50/20 split as devs upgrade): ARR ~$300k, gross ~$210k

Hard constraints

  • MIT-licensed source. omp-deck is open source under MIT. The dashboard and control plane are not — only the tenant-image source. The product is the hosting, not the bits.
  • Exit path is first-class. Volume snapshot download in V0. If customers cannot leave, the convenience pitch is dishonest.
  • No inference gateway in V0. Tenant VMs call vendors directly with customer's BYOK key. Observability-as-paid-feature is Phase 1+.
  • Solo+AI build capacity. ~4–5 weeks of pacing-pace work on top of inheriting AgentVolt's chassis. Sequencing matters: AgentVolt V0 ships first, omp-deck V0 starts after Hermes has operated for ~1 quarter on the chassis.

Trust model — agent VM is RCE-by-design

omp's skills can shell out, install packages, and invoke arbitrary tools; the agent VM is a controlled remote-code-execution surface from boot. Identical posture to AgentVolt's Hermes VM. §7 reuses AgentVolt's three-boundary defense (Cloud Firewall + Tailscale ACLs + per-container cgroups) verbatim.

BYOK-specific implication: the customer's vendor key lives in the tenant .env on the encrypted Cloud Volume. A sufficiently-prompted agent on its own VM can read its own .env; blast radius equals the customer's vendor account. This is disclosed at signup and on the dashboard's BYOK page; it is the cost of giving the customer their own vendor relationship instead of a managed key. See §9.2.

5. Critical flows

Five flows carry over from AgentVolt V0 mechanically. Two are new for omp-deck. The carry-overs get one-paragraph summaries; references point at the AgentVolt §X for the full detail.

5.1 New tenant provisioning inherited

Same state machine and Hetzner Action ID flow as AgentVolt V0 §5.1: pending → volume_created → vm_created → attached → ip_registered → live, idempotent on stripe_event_id, Cloud Network assignment, single-use Tailscale auth key minting, bootstrap-token-based first-boot config exchange. Total signup → live: ~45–60s.

omp-deck-specific: bootstrap exchange populates tenant .env with deck-specific keys (DECK_SESSION_PASSWORD, theme, OMP_AGENT_DIR=/var/lib/omp-deck/agent, initial messaging-bridge config if the customer entered Telegram tokens at signup). Customer's BYOK vendor key is not in the bootstrap payload — it's set post-provision via the dashboard's Providers page (§5.X below).

5.2 Browser launch — auth-bridge sidecar inherited

Identical to AgentVolt V0 §5.2. Dashboard mints 30s-TTL JWT, 302 to {tenant}.ompdeck.app/auth/launch?t=<jwt>, Caddy forwards in X-OmpDeck-Launch header, sidecar verifies against ompdeck pubkey + tenant match, POSTs DECK_SESSION_PASSWORD to local deck /auth/launch, extracts session cookie, returns Set-Cookie upstream. Caddy holds no per-tenant secrets; blast radius of one compromised tenant VM is one tenant.

What omp-deck has to build: the /auth/launch route in apps/server. Today the deck assumes localhost trust; this route accepts a sidecar-minted session POST, sets the deck session cookie, and 302s to /. Maps to AgentVolt's /api/auth/login in shape; deck-specific in implementation.

5.3 Provider key configuration new

The product surface that replaces AgentVolt's inference gateway. Customer adds their vendor keys through the dashboard; the dashboard validates them with a test call, then writes the key to the tenant .env via the control plane.

  1. Customer clicks Providers → Add Anthropic key in the dashboard
  2. Customer pastes API key. Dashboard sends POST /providers/test over mTLS to control plane with {provider, key, tenant_id}; control plane proxies a minimal test call (messages.create with max_tokens=1) to verify the key. Response includes rate-limit headers (anthropic-ratelimit-requests-limit, anthropic-ratelimit-tokens-limit) parsed and stored for dashboard display.
  3. On success: control plane writes the key into the tenant's .env over Tailscale operator path, sends SIGHUP to the deck process; the deck reloads config without restart.
  4. Dashboard shows "Anthropic connected · 1M ITPM · 50 RPM" with vendor-side limits visible to the customer.
  5. Customer can swap providers (Anthropic → OR → OpenAI) without VM restart. The deck's model picker honors whichever provider is configured.

Vendor-side rate-limit surfacing: when the tenant agent receives a 429 from a vendor, the deck surfaces the response headers into the dashboard's Providers page ("Anthropic returned 429 at 14:32, retry-after 60s, current usage 92% of TPM"). This is the customer's problem — their key, their cap — but the dashboard is where they'll look for the diagnosis. Build the surface; don't paper over the 429.

5.4 Messaging webhook inherited

Same shape as AgentVolt V0 §5.4. Webhook hostnames under *.webhook.ompdeck.app (separate zone from launch, both on the multi-SAN cert). Platform → CF → proxy farm → backend VM. Webui verifies platform signature, processes message, agent responds. Outbound long-poll dialed from tenant VM via egress allowlist.

omp-deck-specific: the Telegram bridge runs as a supervised subprocess inside the tenant VM (apps/bridges/telegram, standalone Bun process). It talks to the deck like any other REST + WS client. The dashboard form for Telegram setup writes TELEGRAM_BOT_TOKEN and TELEGRAM_ALLOWED_USER_IDS to the tenant .env; BridgeSupervisor starts the subprocess on next deck restart (or on SIGHUP if hot-reload is wired).

5.5 Auto-repair inherited

Identical to AgentVolt V0 §5.5. Dependency-free /health endpoint on the deck server returns 200 if the Bun process is up and the SQLite DB is reachable (does NOT touch Redis, PG, or vendor APIs — that's the dep-free contract). Same stampede guards (sanity ceiling at 5% fleet, per-tenant 1 recreate/hr, daemon heartbeat).

The deck's existing dev-mode idle reaper for stale sessions stays; it's orthogonal to the daemon-level repair.

5.6 Force-update (CVE-class only) inherited

Identical to AgentVolt V0 §5.6. Audit-logged, 24h email notice (bounce-aware), opt-out via signed-token link, N=20 concurrent paced push, pre-upgrade snapshot, restart-old-on-failure fallback. Total downtime per tenant: ~15s clean, ~2–3min with snapshot restore.

omp-deck-specific: "current-minor-minus-2" advisory emails go out every 90 days for non-CVE updates (AgentVolt V0 Q6 RESOLVED pattern). Customers pin deck versions in the dashboard.

5.7 Decommission flow (Stripe cancellation) inherited

Identical to AgentVolt V0 §5.7. 30-day grace, snapshot at grace end, VM deleted, Tailscale node revoked, volume retained 60 days after VM delete (= day 90 post-cancellation, aligned with R2 final-snapshot retention), volume + R2 prefix wiped at day 90, GDPR right-to-deletion collapses the schedule to immediate.

omp-deck-specific: at cancellation, the dashboard surfaces a "Download your snapshot before day 90" banner with one click to the §5.8 export flow. The exit path is most relevant exactly at cancellation.

5.8 Exit-path snapshot download new

The flow that makes "MIT + hosted for convenience" honest. Anyone can leave with their data intact and run the same MIT-licensed deck against the downloaded volume.

  1. Customer clicks Account → Export → Download Snapshot
  2. Customer enters an export password (chosen in-the-moment; never stored anywhere — Argon2-id parameters surfaced for transparency)
  3. Dashboard POST /api/decks/[id]/export to control plane (mTLS, idempotent on export_request_id)
  4. Snapshot Export service on host B:
    • Fetches tenant's master-key-wrapped DEK from PG
    • Unwraps DEK with the control-plane master key
    • Derives a key from the export password via Argon2-id (parameters: m=64MiB, t=3, p=1; salt persisted alongside ciphertext)
    • Re-wraps the DEK with the password-derived key, produces a wrapped-dek.bin blob
    • Signs a 15-minute-TTL R2 GET URL for the most recent daily snapshot tar
    • Returns {snapshot_url, wrapped_dek_blob, dek_format_version} to the dashboard
  5. Browser downloads both files (snapshot tar + wrapped DEK)
  6. Dashboard surfaces a one-click link to the self-host runbook at docs.ompdeck.app/self-host-from-snapshot

The runbook (docs/self-host-from-snapshot.md in the public MIT repo) walks through: install Docker → derive the DEK with the export password → decrypt the snapshot tar to a host volume → docker compose up against the deck image with the volume mounted → set BYOK keys in .env → done. ~10 minutes if the customer follows it.

The runbook is V0, not Phase 1. Without it the download button is theater. Customers will judge the convenience pitch by how hard it is to leave; making it easy is the credibility move.

Audit + rate-limit: exports logged with customer ID, request ID, timestamp. Rate-limited to 5 exports per tenant per 24h (covers password fat-finger; prevents a compromised dashboard session from exfiltrating snapshot URLs at industrial pace).

6. Operational invariants

Same shape as AgentVolt V0 §6, minus the billing-exactly-once invariant. Verifying each is a V0 ship gate.

  • Customer data preservation across any infrastructure event. State on the Cloud Volume persists independently of VM lifecycle. End-to-end verified by §5.7 restore dress rehearsal.
  • Exit path is real. The §5.8 export flow + self-host runbook is exercised by a canary on the same cadence as the restore drill — a synthetic tenant exports their volume, decrypts it with the export password, runs docker compose up against the resulting state, and confirms the deck boots with tasks/routines/sessions intact. Weekly. new vs AgentVolt
  • Stable URL across VM rebuilds. Same internal-IP-survival sequencing and IP-collision canary as AgentVolt V0 §6. Realistic recreate time 60–90s.
  • Repair completes in <90s, unattended, with stampede guards. Same dep-free probe, sanity ceiling, per-tenant rate limit, daemon heartbeat.
  • No customer secret in cloud-init. 5-minute single-use bootstrap token; customer's BYOK vendor key set post-provision via the §5.3 flow, never via cloud-init.
  • Blast radius is one tenant. Cloud Firewall default-deny lateral + egress allowlist + Tailscale ACLs. A compromised tenant VM reaches one customer's vendor account (per the BYOK blast-radius disclosure in §2 / §9).
  • Master key is recoverable. Off-VM 1Password escrow + two-key rotation procedure, identical to AgentVolt V0 §9.2.
  • Subscription state matches Stripe webhook delivery. Stripe events are idempotent on event.id; failed deliveries retry with exponential backoff via Stripe's own machinery. PG mirrors subscription state but is not the source of truth. replaces billing-exactly-once
  • Customer can pin a version forever. Image registry retains every shipped version with ≥1 active pin (auto-archive at 0 pins after 6 months). Force-push reserved for CVE-class, audit-logged, customer-emailed, opt-out via signed link.
  • Every alert has a runbook. Same discipline as AgentVolt V0 §8.2.

7. Networking & isolation

Identical to AgentVolt V0 §7. The agent VM is RCE-by-design; containment lives at the network layer.

  • Cloud Network sharding across N Networks; tenants.network_id records the assignment (§7.1).
  • Cloud Firewall default-deny lateral + egress allowlist (§7.2). omp-deck's allowlist: api.anthropic.com, openrouter.ai, api.openai.com, Tailscale, Hetzner metadata, GHCR, R2, configured messaging endpoints, configured customer webhooks.
  • Tailscale operator overlay (§7.3). Tags: tag:tenant, tag:ops. ACL: tag:ops → tag:tenant only. Ephemeral single-use auth keys per provision.
  • cgroups + container hardening (§7.4). --memory=3g --memory-swap=3g --cpus=1.5 --pids-limit=512 --read-only on CX23; proportional on CX33. Tenant state on mounted Cloud Volume only.
  • Wildcard cert mechanics (§7.5). Two SANs (*.ompdeck.app + *.webhook.ompdeck.app) on one multi-SAN cert, two DNS-01 challenges, dedicated cert-renewal host.
  • Vercel → control-plane mTLS (§7.6). Same client-cert-per-deployment posture.

9. Secrets management

Same three populations as AgentVolt V0 §9: provisioning-time, at-rest, customer-introduced. The notable shift is that customer-introduced secrets include the vendor API key by default, not opt-in.

9.1 Provisioning-time — no secrets in user-data inherited

Identical to AgentVolt V0 §9.1. 5-minute single-use bootstrap token in cloud-init; real config fetched at first boot from the control plane.

9.2 At rest inherited

Identical to AgentVolt V0 §9.2. Volume encrypted at rest by Hetzner. R2 snapshots client-side-encrypted with per-tenant DEK wrapped by master key. Master key off-VM in 1Password escrow with tested two-key rotation. Tenant .env file mode 0600, owned by the deck user.

9.3 Customer-introduced secrets — BYOK is default

Anthropic / OpenRouter / OpenAI keys live in the tenant .env on the encrypted Cloud Volume, set via the §5.3 dashboard flow. Same as Telegram tokens and other integration secrets: in .env, mode-protected, encrypted at rest.

Blast-radius disclosure (verbatim language for the dashboard's Providers page):

"Your vendor API key lives in your VM's encrypted .env file. omp-deck's agent has shell access to its own VM by design — that's how skills install dependencies and run tools. A sufficiently-prompted agent on your own VM can read your own .env. The blast radius is your vendor account (Anthropic / OpenAI / etc.); it cannot reach other customers' keys or accounts. We never see your key in plaintext after the initial test call. If you want a different posture, you can deny the egress allowlist for skills you don't trust, or run a more restricted skill set."

This is the cost of giving the customer their own vendor relationship instead of a managed key. AgentVolt's customers pay for the comfort of "we hold the keys"; omp-deck's customers pay for the comfort of "managed compute, your account, your spend." Different deals; both honest.

9.4 What's deferred past V0 inherited

Same list as AgentVolt V0 §9.4: KMS as primary master-key store (Phase 2 / SOC2), HSM-backed master key (Phase 3+), just-in-time secret fetching (closes BYOK blast radius on secrets-at-rest), customer-supplied DEK for snapshot encryption, hardware-backed attestation.

Honest posture: V0 is "credible for solo+1 hosting an MIT-licensed deck for tens-to-hundreds of developer customers." Enterprise procurement asks (SOC2, FedRAMP-aligned) are explicitly out of scope; the deferred items above land when those requests come in.

10. Tenant image strategy

The biggest single simplification vs AgentVolt V0. omp-deck source is MIT and authored by the same team that operates the hosting. There is no upstream-fork constraint and no "no upstream PRs in the critical path" rule — the tenant image is the project's own Dockerfile.

10.1 Build & distribution

  • Tenant image built by GitHub Actions on tag push to the omp-deck repo. Tagged ghcr.io/ompdeck/deck:<deck-version>-<sdk-version>-<rev>.
  • The compound version captures the tuple (deck rev, @oh-my-pi/pi-coding-agent version); both matter for pinning because the deck embeds the SDK in-process.
  • Images signed with cosign; bootstrap verifies signature before container start.
  • Version retention: every image with ≥1 active pin retained indefinitely; 0-pin images auto-archive after 6 months. Same as AgentVolt V0 §10.1.
  • No upstream mirror needed (vs AgentVolt's ghcr.io/agentvolt/upstream-mirror) — we ARE the upstream.

10.2 Migration sandbox

Same pattern as AgentVolt V0 §10.2. Major-version migrations run on an ephemeral CX23 against a clone of the tenant's volume. Same Cloud Firewall + egress allowlist.

10.3 Route audit (pre-V0 gate)

Lighter than AgentVolt's because we control the source, but not absent. Before V0 ships, every route in apps/server/src/routes-*.ts is classified:

  • safe-as-is — most surface (tasks, routines, inbox, sessions, marketplace UI)
  • auth-required — anything that mutates state; deck already enforces session-cookie
  • deny-at-proxy — none currently, but the audit may identify some (e.g. arbitrary fs operations not scoped to the workspace)
  • customer-visible config only — /api/settings, /api/providers (new for V0)

Findings recorded in docs/proposals/route-audit.md, replayed against each release.

11. V0 SLA, residency & status

11.1 V0 region

Region target: NBG (Nuremberg, EU) by default, same rationale as AgentVolt V0 §11.1. Open to revisit if the customer pipeline turns out to be majority US (see §13 Q4) — for omp-deck specifically, the developer audience may pipeline more US-heavy than AgentVolt's, since omp adoption is likely US-skewed.

11.2 V0 SLA language (for ToS)

"omp-deck Managed Hosting V0 is a single-region, best-effort hosted service. We do not commit to a formal uptime percentage during V0. Multi-region failover is committed by tenant #2000 or [date], whichever comes first. Inference availability depends on the customer's own vendor account (Anthropic / OpenRouter / OpenAI) — we do not proxy or buffer inference; vendor outages on the customer's chosen provider are visible to the customer directly. omp-deck source is MIT-licensed; customers may export their data and self-host at any time without our involvement (see Export Snapshot)."

The MIT + exit-path language is load-bearing for the convenience-pricing thesis. Stating it in the ToS removes the ambiguity that would otherwise surface as churn after the first customer-facing outage.

11.3 Status surface

Same out-of-band posture as AgentVolt V0 §11.3 (Statuspage.io or Cloudflare Workers static page). Independent of Vercel + PG so customers can see incidents during their failure modes.

11.4 DNS posture

Single Cloudflare account holding both wildcard zones. Documented account-lockout recovery runbook. Secondary nameserver via NS1 / DNSimple deferred to Phase 1 — same as AgentVolt V0.

13. Open questions for review

Q1 · Dashboard ownership of provider keys at-rest
Today's design has the control plane store nothing about the customer's vendor key after the §5.3 test call (write-through to tenant .env, forget). Alternative: store a per-tenant fingerprint (last-4-chars, hash) for dashboard display so the customer can see which key is active without us reading the key. The fingerprint approach is friendlier UX; confirm we want it before locking the write-through-and-forget posture.
Q2 · Telegram bridge supervision under cgroup limits
The Telegram bridge is a separate Bun subprocess (apps/bridges/telegram) supervised by the deck's BridgeSupervisor. Under --pids-limit=512 + --memory=3g, two concurrent bridges (Telegram + Discord when that ships) plus the deck server plus the omp SDK process plus skills' subprocess shells — does that fit? Worth measuring against a real workload before shipping POWER-tier "unlimited bridges."
Q3 · PWA scope at V0 vs Phase 1
Desktop PWA (installable, offline shell, push from browser) is one week of work and unlocks the "always-on agent" pitch. Mobile push (real iOS / Android notifications) is a second week and requires a push vendor (OneSignal / Firebase). V0 = desktop PWA only, or both?
Q4 · Customer pipeline geography
Same question AgentVolt's §13 Q12 raises. omp-deck's audience (developers using omp) likely skews more US than AgentVolt's audience. If pipeline is majority US, ASH should be the V0 default and EU triggers Phase 2 — opposite of AgentVolt's default.
Q5 · Brand and domain
Working assumption is ompdeck.app + webhook.ompdeck.app. Confirm the brand name before cert provisioning — DNS-01 wildcards take a few hours to provision and the domain choice is sticky. Alternatives considered: deck.omp.app, hosted.omp.app, app.ompdeck.com.
Q6 · Volume layout authoritative spec
Before snapshot tooling can target the right paths, every persistent location the deck writes to has to be enumerated. From docs/architecture.md: SQLite DB (tasks, routines, routine_runs, inbox_items, ...), ~/.omp/agent/ (sessions JSONL, auth.db, marketplaces.json, installed_plugins.json), Telegram bridge SQLite (per-chat session map), workspace dir, tenant .env. Anything else the deck or its skills can write? Worth a one-pass inventory before V0 freezes the volume layout.
Q7 · OSS deck divergence over time
The hosted tenant image diverges slightly from the OSS deck (auth-bridge sidecar contract, dep-free /health guarantees, boot-time config consumer). Are those changes upstreamed to the OSS deck so any self-hoster gets them, or do they live only in the hosted image overlay? Upstreaming keeps the MIT story honest (the same code runs hosted and self-hosted); overlay-only keeps hosted-specific complexity out of the OSS surface. I'd vote upstream — the auth-bridge route in particular is useful to any self-hoster who wants to put the deck behind their own JWT-minting auth proxy.
